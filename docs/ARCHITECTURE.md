# Архитектура платформы СМУ ФНИСЦ РАН

## Цель

Портал строится как управляемая ИТ-платформа, а не статический сайт. Его задача — одновременно решать публичные, административные и карьерно-навигационные задачи: показывать состав Совета, новости, мероприятия, публикации и СМИ; автоматизировать пресс-релизы; считать научные показатели; помогать членам СМУ понимать следующий карьерный шаг.

## Слои

Frontend: Next.js App Router, серверные компоненты для публичных разделов, API routes для автоматизации и интеграций. Дизайн опирается на строгую академическую айдентику: тёмно-синий, красный акцент, светлый фон, карточная структура, крупная типографика. Точные цвета можно заменить после снятия палитры с официального логотипа ФНИСЦ.

Data: Prisma schema покрывает участников, идентификаторы, публикации, авторство, наукометрические снимки, медиаупоминания, события, новостные черновики, фото, педагогический стаж, карьерные override и меры поддержки.

Local public-data layer: для локального релиза harvest пишет воспроизводимые JSON-витрины в `data/public` и `public/generated`. Публичные страницы читают этот слой напрямую, поэтому сайт запускается даже без production worker/CI и честно показывает частичные статусы источников.

Workers: в MVP это `scripts/*` и защищённые API routes. В production лучше выделить отдельный cron/worker, который запускает сборщики ночью, кеширует результаты и пишет отчёты аудита.

Integrations: eLibrary/РИНЦ, Scopus, Web of Science, OpenAlex/ORCID/Crossref, Google News RSS, sitemap, Telegram public pages, LLM API. Все ключи только в environment variables.

Moderation: автоматические черновики новостей и карьерные подсказки не публикуются без редакторской проверки. Система должна показывать `facts_to_verify`, журнал аудита и статус доверия источника.

## Поток публикаций

1. Участник СМУ имеет `Member` и набор `Identifier`: `elibrary_author_id`, `spin`, `orcid`, `scopus_author_id`, `wos_researcher_id`.
2. Сборщики получают исходные записи из eLibrary/РИНЦ, Scopus, WoS и OpenAlex.
3. `mergePublicationSets` строит канонический список по eLibrary item id, DOI, нормализованному названию и году.
4. Для публикаций вычисляются источники, цитирования, признаки ВАК, тип публикации и возможные специальности.
5. `buildScientometricSummary` считает агрегаты для участника и Совета.
6. Career engine использует этот слой для milestone.

## Поток новости

1. Редактор загружает программу мероприятия PDF/DOCX и фото.
2. `extractTextFromUpload` извлекает текст.
3. `generatePressReleaseDraft` выбирает LLM-провайдера или локальный fallback.
4. Модель возвращает JSON: `title`, `lead`, `body_html`, `photo_captions`, `facts_to_verify`.
5. Администратор проверяет факты и публикует.

## Поток карьеры

1. Система получает публикации, степени/звания, педагогический стаж и ручные подтверждения.
2. `evaluateCareer` сравнивает профиль с правилами из `data/rules/career-rules.json`.
3. Пользователь видит не длинный нормативный текст, а карту: что пройдено, что почти готово, чего не хватает.
4. Руководитель СМУ видит heatmap/дашборд Совета: кому нужна публикация, кому — учебное издание, кому — консультация по защите.

## Production-доработки

Нужны полноценная авторизация, хранение фото в S3/MinIO, модерация черновиков в админке, очередь задач, отчёты качества парсинга, юридически выверенные правила карьерной карты, импорт актуального перечня ВАК и раздельные права доступа к приватным контактам.
